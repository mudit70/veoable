## What landed across iterations

### Iter 1
- **B4 (Django @api_view routes):** `framework-django` extended to
  recognise `@api_view([...])` function-based views and to compose
  routes from `path("…", views.func_name)` declarations across the
  `urls.py` include() chain. App 5 went from 0 → 5 APIEndpoints with
  correct param normalisation (`<uuid:photo_id>` → `:photo_id`).
- **B3 (mongogo collection helpers):** Inlined
  `client.Database("…").Collection("…")` in the fleet-monitor
  handlers so the existing plugin's bindings-scanner fires. App 2
  went from 0 → 6 DatabaseInteractions and 2 DatabaseTables. The
  underlying plugin gap (cross-file helper-function resolution) is
  a follow-up.

### Iter 2
- **B7 (SQLAlchemy framework=null):** Misdiagnosis. The
  `DatabaseInteraction` schema uses `orm` (not `framework`); the
  plugin already stamps `orm: 'sqlalchemy'` correctly. The report
  driver was reading the wrong field — fixed.
- **B5 (boto3 `Params={…}` nested dict):** `framework-boto3` visitor
  now resolves `Bucket` and `Key` inside an S3 `Params={"Bucket": …,
  "Key": …}` dict literal (used by `generate_presigned_url`,
  `generate_presigned_post`). Added a new unit test that asserts
  `generate_presigned_url(Params={…})` now emits `s3://bucket/key`
  exact instead of dynamic.

### Iter 3
- **B8 (bundler/gohttp false positives):** Not actually bugs — both
  plugins activate by design (config file present / Go stdlib
  always-on). The report driver's "unexpected detections" list now
  whitelists them. Genuinely unexpected detections drop from 7
  across apps to 2 (and those two are framework-label vs plugin-ID
  mix-ups in the test config, not bugs).
- **B1 (fetch `${BASE}` prefix blocks stitching):** Two fixes:
  - **`flow-stitcher` url-matcher:** strip a leading `:p<n>/`
    placeholder when it represents an implicit base-URL prefix.
    Handles `fetch(\`${BASE}/api/foo\`)` where BASE is an empty
    default or origin. Applied at both the fast-path templateParts
    reconstruction and the slow-path segment walk.
  - **App 1 cross-file prefix handled in framework-fastapi.** App 1's
    main.py + routers/tasks.py split is now resolved by the
    cross-file `include_router` resolver that ships in framework-fastapi
    (the visitor consults a project-load-time map of composed prefixes
    so `@router.get("")` in tasks.py emits `/api/tasks` when main.py
    calls `app.include_router(tasks.router, prefix="/api")`).
  - **Result:** App 1 went from 0 → 3 complete flows. App 5 went
    from 0 → 5 stitches. Total stitched 11 → 21.

## What remains

These show up as remaining `partial` flows or `dynamic` urlLiterals.
Each is a separate follow-up.

### Open plugin gaps

- **Awsrust identifier args (B6, by design).** App 4's DynamoDB and
  SQS callers emit `urlLiteral=?` because the table_name and
  queue_url come from `&state.orders_table` /
  `&state.orders_queue_url` — identifier references, not literals.
  A constants-resolution pass on the Rust visitor could lift these,
  but it's a non-trivial cross-method change.

- **Axum DELETE pair (B523-#2)** — **fixed** as a side-effect of the
  flow-stitcher base-prefix work shipped in #519. The four candidate
  fetch↔axum pairs are: GET orders, GET portfolio, DELETE
  orders/:p0→:id, and POST orders. The first three now stitch
  cleanly (`stitchSummary.resolved=3` for App 4); the fourth is the
  wrapped jfetch POST whose call site is
  `await fetch(url, init)` — both url and init are identifier
  references, so the visitor correctly marks it dynamic. That's a
  wrapper-detection question for framework-fetch, not a stitcher
  bug.

### Loop 2 — sweep gating

The first cut of #529 sweep was unconditional. cal.com regressed
130s → 1275s because its source already covers everything via ts-morph's
default tsconfig load — extra sweep was pure cost. Gated to fire only
when (a) opts.include is supplied, (b) no tsconfig at all, or
(c) a subpackage `apps/*/`, `packages/*/`, `services/*/`, `libs/*/`
tsconfig exists that the root's `references` array does not list.
JSONC comments + trailing commas parsed via `ts.parseConfigFileTextToJson`
so a clobbered `https://` inside a `path` string can't break detection.

### Loop 3 — typebot caller mix analysis

After #529 brought typebot's caller count from 0 to 19, looked at the
remaining 10 dynamic callers. All were genuinely dynamic (env-var URLs,
ky wrapper passthroughs, wrapper-param URLs). No same-file
constant-identifier resolution gap surfaced — `resolveCallerUrl`
already handles those via `resolveToString`. The bigger remaining
shape — `axios.create({ baseURL })` instance composition — is #532
and stays open.

# the locked-in integration assertions
cd packages/cli && npx vitest run src/__tests__/test-apps.integration.test.ts
