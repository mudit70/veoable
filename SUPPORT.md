# Getting Help

Thanks for using Veoable. Here's where to go depending on what you need.

## Questions & discussion

- **GitHub Discussions** — <https://github.com/mudit70/veoable/discussions>.
  Best for open-ended questions, "how do I…?", showing off a plugin you
  built, or sharing an analysis workflow.
- **Framework matrix / plugin capabilities** — start with the
  [User Guide](docs/userguide.md). It lists every supported framework
  and what the analyzer emits for it.
- **MCP setup** — the
  [LLM Client Install Guide](docs/llm-client-install.md) covers Claude
  Desktop, Cursor, Continue, and Zed.

## Bug reports

Open a
[new issue](https://github.com/mudit70/veoable/issues/new/choose)
using the **Bug report** template. Please include:

- The exact command or invocation.
- A minimal reproducer (repo link + branch is ideal).
- Expected vs. actual output.
- Veoable / package version and Node.js version.

## Feature requests

Open a
[new issue](https://github.com/mudit70/veoable/issues/new/choose)
using the **Feature request** template. Framing the request in terms of
the analysis problem you're trying to solve helps us pick the right
layer to add support at (language plugin, framework plugin, flow
stitcher, CLI, or MCP).

## Security issues

**Do NOT open a public issue.** See [`SECURITY.md`](./SECURITY.md) for
the private disclosure process.

## Commercial support

There is currently no paid support tier. If your organization is
interested in one — reach out via `hello@veoable.dev` and let us know
what you'd need.

## Response expectations

Veoable is maintained by a small team. We aim to acknowledge issues
within a few business days. Security reports have a stricter SLA — see
[`SECURITY.md`](./SECURITY.md).
